### ADR-0007 — listPipeline needs-attention ordering (deterministic) + live-board filter
- Date: 2026-07-13
- Zone: YELLOW
- Context: spec/US-4 requires the board "sorted so the ones needing attention appear first" but does not define the signals or their precedence; tasks.md flags the choice as the implementer's call, deterministic and documented.

- Reversibility: easy — one ORDER BY clause + one WHERE filter in `listPipeline`, pinned by tests that can be re-pinned.
- Article-safety: confirmed no Article I–IX violation (deterministic SQL, Article III; analytical joins only, Article IV).

### ADR-0010 — saveIdentity records consent itself; erasure withdraws it
- Date: 2026-07-13
- Zone: YELLOW
- Context: T008a requires saveIdentity to refuse "unless the buyer row has consent_flag=1 or the call itself records consent — pick one semantic". Real trade-off: a pre-existing-flag gate means a separate consent write must happen first (two calls, a window where the flag is set but no identity exists, and nothing ties the flag to a date); call-records-consent makes the identity write and the consent record one atomic act.
- Decision: saveIdentity RECORDS consent — it requires a valid ISO consent date (throws otherwise, before any write) and, in one transaction, sets buyers.consent_flag = 1 and upserts the encrypted identity row with consent_date. There is therefore no code path that creates an identity row without recorded consent (Article IV / FR-14 holds by construction). Corollaries: (a) eraseIdentity resets consent_flag to 0 — erasure is treated as consent withdrawal, so the flag never claims consent for identity that no longer exists, and a re-save needs fresh consent; (b) eraseIdentity deliberately needs NO encryption key, so key loss can never block a GDPR erasure; (c) key format is 32 bytes as base64 (44 chars) or hex (64 chars) in CONSTRUCTOR_PII_KEY, fail-secure (throw naming the var, never a fallback), documented in .env.example with an openssl generation command.
- Alternatives considered: pre-existing consent_flag gate (rejected: consent capture and identity capture happen in the same operator moment — splitting them adds a failure window and double-entry friction, Article I/VII, with no privacy gain); keeping consent_flag = 1 after erasure (rejected: a truthy consent flag with no identity row misstates the GDPR position and would let a later save path skip re-consent).
- Reversibility: easy — the consent check and flag updates are localized to src/db/identity.ts and pinned by tests/identity.test.ts; switching semantics is a small, test-visible change while the table shape (locked, data-model.md) is untouched.
- Article-safety: confirmed no Article I–IX violation (IV strengthened: consent is atomic with identity, PII never in errors/logs, analytics proven queryable after erasure; II/V untouched; VIII: node:crypto built-in, no new deps).

- Date: 2026-07-13
- Zone: YELLOW
- Context: T008a leaves open (a) what "recorded consent" means for saveIdentity, (b) the accepted encoding of the 32-byte CONSTRUCTOR_PII_KEY, (c) the exact byte layout of the encrypted BLOB, and (d) upsert semantics for a re-saved identity.

- Date: 2026-07-13
- Zone: YELLOW
- Context: tasks.md fixes the five routes and "reject empty next_action (400)" but leaves open the request/response JSON shapes, the error body format, and which failures are 400 vs 404 (ZONING Step 4 names "input shapes" explicitly). (f) makeServer(db, port = 0) binds 127.0.0.1 and is the ONLY export; import.meta.main starts the dev server (initDb() + PORT env, default 3000 — a local dev port, not a secret, so a default is not fail-open).

- Reversibility: easy — contract lives in one file (src/api/server.ts) and is pinned by tests/api.test.ts; T012 (web) is the only future consumer and lands after this.
- Article-safety: confirmed no Article I–IX violation (II: blank nextAction 400 at boundary with queries guard + SQL CHECK behind it; III: no LLM, no network egress — server only listens on localhost; IV: PII refused at the boundary, never stored or echoed beyond the offending request; VIII: Bun.serve directly, zero deps, one module).

- Context: T010 says "response includes suggested counter when applicable" but leaves open (a) which layer computes it, (b) which unit's asking_current applies when the offer body omits unitId (units are optional on offers), and (c) what happens when no unit is in play or the stored asking is unusable.
- Decision: (a) `logOffer` in queries.ts returns `LogOfferResult = LogEventResult & { counter: CounterSuggestion | null }` — queries.ts owns the one SQL read (asking_current) and calls the pure domain `counter()` (ADR-0003 math untouched); server.ts needs ZERO changes because it already serializes the queries result verbatim, staying a thin validator, and T012 (web) gets the same suggestion for free. (b) The effective unit is the offer's own unitId, else the opportunity's existing focus unit — read via `focus_unit_id` AFTER the write transaction, which by the ADR-0014 COALESCE mirror holds exactly `COALESCE(event unit, prior focus)`; an operator who set the unit at the viewing isn't forced to re-enter it at the offer (Article I). (c) `counter: null` (explicit field, never absent) when the offer is at/above asking, when no unit is in play, or when the stored asking is non-positive — the counter is ADVISORY and must never fail a capture that already committed, so a data-quality issue in units yields no suggestion instead of a 500 after a persisted write.
- Alternatives considered: computing the counter in server.ts via a `getUnitAsking` query (rejected: pushes orchestration into the file whose contract is shape-validation only, and the web client would need to duplicate the call); counter only when the offer body carries unitId (rejected: punishes the Article-I-correct flow where the unit was captured once at the viewing); omitting the field instead of null (rejected: an explicit null distinguishes "no counter applies" from "old server without T010" for the T012 client); letting a non-positive stored asking throw RangeError (rejected: the event row is already committed — erroring would report failure for a successful capture).
- Reversibility: easy — one result field, one read, one guard in `logOffer`, pinned by five T010 tests in tests/api.test.ts (below/at/above asking with exact 300000/270000→288000 numbers, focus-unit fallback, no-unit null, Article II unchanged).
- Article-safety: confirmed no Article I–IX violation (III: counter math stays the locked deterministic ADR-0003 formula in src/domain/counter.ts — this task only calls it; II: blank next_action on an offer still 400s with nothing written, pinned; IV: read touches units/opportunities only; VIII: no new deps, no new module).

### ADR-0019 — updateAskingPrice input/result shape, validation, and no-next_action semantics (T010a)
- Date: 2026-07-13
- Zone: YELLOW
- Context: T010a fixes the behavior (asking_current update + exactly one price_changes row, atomically) but leaves open (a) the exact call/result shape, (b) validation rules for newPrice/reason, and (c) how the src/db per-write checklist's `assertNextAction` line applies to a write that creates no opportunity/event.
- Decision: (a) Follows the layer convention: `updateAskingPrice(db, { unitId, newPrice, reason?, at? })` returning `{ unitId, oldPrice, newPrice, changedAt }` — oldPrice comes back so UI/report callers can confirm the delta without a second read; `at?` keeps the ISO `changed_at` deterministic in tests. (b) newPrice must be a positive integer € (RangeError, mirroring logOffer's amount rule) checked BEFORE any DB statement; reason is optional (schema column is nullable) and stored NULL when absent; a same-price call is NOT special-cased — it still logs one row (the operator's action is a fact worth auditing, and rejecting it adds a rule the spec never asked for). The old price is read INSIDE the transaction, so the logged old_price can never race the update. `asking_initial` is never touched (realization baseline, v_price_realization). (c) `assertNextAction` is deliberately NOT called: Article II governs opportunities/events ("no opportunity without a next action") and this write touches neither; the price-update signature in tasks.md/data-model.md has no next_action, and price_changes has no such column. `assertNoPiiKeys` IS called first, keeping the layer-wide Article IV guard universal. Atomicity is pinned by a test that forces the log INSERT to fail via a SQLite trigger and asserts the price update rolled back (both-or-neither).
- Alternatives considered: positional `updateAskingPrice(db, unitId, newPrice, reason)` matching the brief's shorthand literally (rejected: every other write in queries.ts takes an input object with `at?` — divergence costs more than it buys); requiring a non-empty reason (rejected: schema says nullable, and Article I favors fewer required fields — a forced reason breeds junk text); rejecting newPrice === current (rejected: adds an unrequested invariant; the audit row keeps the action visible either way); returning void (rejected: callers would immediately re-query for save feedback, Article I).
- Reversibility: easy — one function + one interface pair in queries.ts, pinned by seven T010a tests in tests/constraints.test.ts; no schema change (price_changes existed verbatim in data-model.md).
- Article-safety: confirmed no Article I–IX violation (II: not weakened — no opportunity/event row is written or restamped by this path, guards on all other writes untouched; III: no wall-clock beyond the standard `nowIso(at?)` write-path convention, math is arithmetic-free logging; IV: analytical tables only, PII-key guard first; V: no location fields involved; VIII: no new deps, same module).

### ADR-0020 — Seed file shape: declarative natural-key snapshot, validate-all-then-one-transaction, insert-if-absent idempotency (T011)
- Date: 2026-07-13
- Zone: YELLOW (ZONING Step 4 names "seed JSON shape" explicitly)
- Context: US-7/FR-10 require a Day-0 migration from "a structured input file" but no spec doc fixes the JSON shape, the reference mechanism between entities, re-run semantics, or how declared stages/temperatures relate to the capture-path derivation rules.
- Decision: (a) Shape = three top-level arrays — `projects` (units nested), `buyers` (analytical only), `opportunities` (sales_events history nested) — with snake_case field names matching the data-model columns 1:1; cross-references use NATURAL KEYS (`project` = project_name, `buyer` = pseudonym, `unit`/`focus_unit` = unit_code within the opportunity's project), never raw integer ids, so a hand-authored migration file cannot corrupt id allocation. (b) The file is a DECLARATIVE snapshot of an already-existing pipeline: stage/temperature/next_action/updated_at are stored as stated (migrated facts) — seed does NOT re-derive temperature or replay forward-only stage floors; the derivation rules govern captures, not migrations (the domain/queries code is untouched). (c) Loadable event types are the Phase-A capture universe only (inquiry|viewing|offer) — a seed file cannot smuggle in reservation/contract history (Phase B stays dark). (d) Validation is TOTAL and runs BEFORE any DB statement: strict unknown-key rejection per entity (a PII-shaped key — same regex as the queries-layer guard — fails with an Article IV message; buyers are pseudonyms-only since the example file is committed), Article II blank-next_action checks on every opportunity AND event, Article V non-empty micro_area, enum/ref/grain (one opportunity per buyer↔project, in-file too) checks; then ONE transaction loads everything — a bad record anywhere means nothing is written. (e) Idempotency = insert-if-absent by natural key (projects on builder+name, units on project+code, buyers on pseudonym, opportunities on the grain); an existing opportunity skips its nested events too (its history came with it the first time) — re-seed never crashes, never duplicates, never overwrites live pipeline state. (f) A `"#N"` pseudonym loads as buyer id N (explicit id inside the transaction), preserving the ADR-0014 pseudonym↔id invariant so a later createLead (MAX(id)+1) can never mint an already-taken pseudonym; non-#N pseudonyms take auto ids. (g) CLI (`bun run seed <file.json>`) prints a Greek summary/error framing (operator surface) while validation Error messages keep the layer convention (English detail with Article prefixes — the HTTP API remains the full Greek translation boundary, ADR-0017). (h) No wall-clock anywhere: every timestamp comes from the file, so seeding is fully deterministic. consent_flag is always 0 — identity/PII is out of scope for seed by construction.
- Alternatives considered: replaying the file through createLead/logViewing/logOffer (rejected: captures DERIVE stage/temperature and stamp their own semantics — a migrated pipeline's declared state would be unreproducible, pseudonyms would be regenerated, and Lead-history would be faked for on-the-spot opportunities); raw-id references (rejected: hand-authored ids are the classic silent-corruption vector and make idempotency ill-defined); upsert/overwrite on re-seed (rejected: an accidental second run silently mutating live pipeline rows is worse than skipping — "does not crash or duplicate" is the requirement, not "sync"); lenient unknown-key tolerance (rejected: a typoed field name would silently drop migrated data, and PII keys must be loudly refused, not ignored).
- Reversibility: easy — one loader module (src/db/seed.ts) + one committed example file, pinned by 13 tests in tests/seed.test.ts; the shape can still evolve before T020 since no external system emits it yet.
- Article-safety: confirmed no Article I–IX violation (II: blank next_action anywhere in the file rejected before any write, SQL CHECK as backstop; III: no LLM, no network, no wall-clock — file-driven timestamps only; IV: PII-shaped keys rejected with a loud error, buyers analytical-only, consent untouched at 0, buyer_identity never referenced; V: micro_area required non-empty at project+micro-area precision; VIII: bun:sqlite direct, zero new deps).

### ADR-0021 — Comps entry shape, entered_by accountability, and the own-sales UNION ALL merge (T011a)
- Date: 2026-07-13
- Zone: YELLOW (ZONING Step 4: input shapes and a merge semantics the spec left open — FR-12/data-model fix the table and the source enum, not the API/CLI shapes or how "own sold units auto-count" is computed)
- Context: T011a needs `addComp` + `bun run comp:add` + the merge query the monthly comparative (T016) will consume. Open choices: input/result shapes, which validations are hard errors, whether entered_by is always required, how own sold units merge with comps rows (dedup? ordering? sale_date for units?), and the CLI argument grammar.
- Decision: (a) `addComp(db, {area, microArea, salePrice, source, sqm?, rooms?, saleDate?, enteredBy?, note?})` mirrors the comps columns 1:1 (camelCase per layer convention), validates ALL before any DB statement, and wraps the insert in the standard transaction; salePrice must be a positive integer € (RangeError, mirroring logOffer/updateAskingPrice) and is documented everywhere — types, errors, CLI help ("ΠΡΑΓΜΑΤΙΚΗ τιμή ΠΩΛΗΣΗΣ — ποτέ ζητούμενη") — as an actual SALE price, never asking. (b) `entered_by` is required ONLY for `manual_known_sale` (manual market data must be accountable to an operator); for `own_transaction` rows it stays optional, matching the schema comment ("operator, for manual entries"). (c) The merge (`compsForMicroArea`) is `comps WHERE micro_area = ?` UNION ALL own sold units (`units.sale_price IS NOT NULL` joined through their project's micro_area) auto-labelled `own_transaction` — NO dedup: own sales count automatically (Article VII, never double-entered; the CLI help says "μην τις ξανακαταχωρείς"), and a duplicate would be operator error, not something to silently guess away. Units contribute `sale_date = NULL` (no sold-date column in Phase A; contract-event dates are Phase B) rather than faking one. Ordering is a deterministic ORDER BY over result columns only (compound-SELECT restriction: no expressions) — sale_price, source, sale_date, sqm, rooms — any residual tie is field-identical, so report output is byte-stable (Article III). (d) No `at?`/timestamp: comps has no created_at column and the write derives nothing from a clock — `sale_date` is operator-supplied data (per-write checklist line N/A by design, ADR-0019 precedent). (e) CLI grammar `--area --micro-area --price --source --entered-by [--sqm --rooms --date --note]` via node:util parseArgs (builtin, strict — unknown flag = loud Greek-framed error + usage); no args → usage + exit 1; `--help` → usage + exit 0; PII-key guard runs first like every src/db write (Article IV), and `assertNextAction` is deliberately absent (comps is not a capture — RULING 2026-07-13b scope).
- Alternatives considered: always-required entered_by (rejected: schema marks it "for manual entries" and seed/report-generated own_transaction rows have no operator to name); dedup own units vs comps rows in the merge (rejected: no natural key exists — unit_code is not stored on comps — and silently dropping a row the operator entered hides the double-entry mistake instead of surfacing it at review); positional CLI args (rejected: nine mostly-optional fields make flags self-documenting, and Greek micro_area strings with spaces/interpuncts are hostile to positional parsing); synthesizing a unit sale_date from opportunities.updated_at (rejected: that is a pipeline-activity timestamp, not a sale date — a wrong number in a client-facing comparative is worse than NULL).
- Reversibility: easy — one new module (src/db/comps.ts) + one package.json script, pinned by 15 tests in tests/comps.test.ts; no schema change (comps table shipped verbatim in T002); T016 consumes `compsForMicroArea` so the merge semantics can still evolve until CHECKPOINT 2.
- Article-safety: confirmed no Article I–IX violation (II: no opportunity/event written, all capture guards untouched; III: no wall-clock, no LLM, deterministic ordering; IV: PII-key guard before any statement, buyer_identity never referenced; V: micro_area hard-required non-empty with an Article V error, merge keyed on micro_area; VII: own sales auto-count, never re-typed; VIII: bun:sqlite direct, zero new deps).

### ADR-0007 — listPipeline needs-attention ordering (deterministic) + live-board filter
- Date: 2026-07-13
- Zone: YELLOW
- Context: spec/US-4 requires the board "sorted so the ones needing attention appear first" but does not define the signals or their precedence; tasks.md flags the choice as the implementer's call, deterministic and documented.

- Article-safety: confirmed no Article I–IX violation (deterministic SQL, Article III; analytical joins only, Article IV).

### ADR-0010 — saveIdentity records consent itself; erasure withdraws it
- Date: 2026-07-13
- Zone: YELLOW
- Context: T008a requires saveIdentity to refuse "unless the buyer row has consent_flag=1 or the call itself records consent — pick one semantic". Real trade-off: a pre-existing-flag gate means a separate consent write must happen first (two calls, a window where the flag is set but no identity exists, and nothing ties the flag to a date); call-records-consent makes the identity write and the consent record one atomic act.
- Decision: saveIdentity RECORDS consent — it requires a valid ISO consent date (throws otherwise, before any write) and, in one transaction, sets buyers.consent_flag = 1 and upserts the encrypted identity row with consent_date. There is therefore no code path that creates an identity row without recorded consent (Article IV / FR-14 holds by construction). Corollaries: (a) eraseIdentity resets consent_flag to 0 — erasure is treated as consent withdrawal, so the flag never claims consent for identity that no longer exists, and a re-save needs fresh consent; (b) eraseIdentity deliberately needs NO encryption key, so key loss can never block a GDPR erasure; (c) key format is 32 bytes as base64 (44 chars) or hex (64 chars) in CONSTRUCTOR_PII_KEY, fail-secure (throw naming the var, never a fallback), documented in .env.example with an openssl generation command.
- Alternatives considered: pre-existing consent_flag gate (rejected: consent capture and identity capture happen in the same operator moment — splitting them adds a failure window and double-entry friction, Article I/VII, with no privacy gain); keeping consent_flag = 1 after erasure (rejected: a truthy consent flag with no identity row misstates the GDPR position and would let a later save path skip re-consent).
- Reversibility: easy — the consent check and flag updates are localized to src/db/identity.ts and pinned by tests/identity.test.ts; switching semantics is a small, test-visible change while the table shape (locked, data-model.md) is untouched.
- Article-safety: confirmed no Article I–IX violation (IV strengthened: consent is atomic with identity, PII never in errors/logs, analytics proven queryable after erasure; II/V untouched; VIII: node:crypto built-in, no new deps).

- Date: 2026-07-13
- Zone: YELLOW
- Context: T008a leaves open (a) what "recorded consent" means for saveIdentity, (b) the accepted encoding of the 32-byte CONSTRUCTOR_PII_KEY, (c) the exact byte layout of the encrypted BLOB, and (d) upsert semantics for a re-saved identity.

- Date: 2026-07-13
- Zone: YELLOW

- Article-safety: confirmed no Article I–IX violation (II: blank nextAction 400 at boundary with queries guard + SQL CHECK behind it; III: no LLM, no network egress — server only listens on localhost; IV: PII refused at the boundary, never stored or echoed beyond the offending request; VIII: Bun.serve directly, zero deps, one module).

### ADR-0020 — Seed file shape: declarative natural-key snapshot, validate-all-then-one-transaction, insert-if-absent idempotency (T011)
- Date: 2026-07-13
- Zone: YELLOW (ZONING Step 4 names "seed JSON shape" explicitly)
- Context: US-7/FR-10 require a Day-0 migration from "a structured input file" but no spec doc fixes the JSON shape, the reference mechanism between entities, re-run semantics, or how declared stages/temperatures relate to the capture-path derivation rules.
- Decision: (a) Shape = three top-level arrays — `projects` (units nested), `buyers` (analytical only), `opportunities` (sales_events history nested) — with snake_case field names matching the data-model columns 1:1; cross-references use NATURAL KEYS (`project` = project_name, `buyer` = pseudonym, `unit`/`focus_unit` = unit_code within the opportunity's project), never raw integer ids, so a hand-authored migration file cannot corrupt id allocation. (b) The file is a DECLARATIVE snapshot of an already-existing pipeline: stage/temperature/next_action/updated_at are stored as stated (migrated facts) — seed does NOT re-derive temperature or replay forward-only stage floors; the derivation rules govern captures, not migrations (the domain/queries code is untouched). (f) A `"#N"` pseudonym loads as buyer id N (explicit id inside the transaction), preserving the ADR-0014 pseudonym↔id invariant so a later createLead (MAX(id)+1) can never mint an already-taken pseudonym; non-#N pseudonyms take auto ids. (h) No wall-clock anywhere: every timestamp comes from the file, so seeding is fully deterministic. consent_flag is always 0 — identity/PII is out of scope for seed by construction.

- Reversibility: easy — one loader module (src/db/seed.ts) + one committed example file, pinned by 13 tests in tests/seed.test.ts; the shape can still evolve before T020 since no external system emits it yet.
- Article-safety: confirmed no Article I–IX violation (II: blank next_action anywhere in the file rejected before any write, SQL CHECK as backstop; III: no LLM, no network, no wall-clock — file-driven timestamps only; IV: PII-shaped keys rejected with a loud error, buyers analytical-only, consent untouched at 0, buyer_identity never referenced; V: micro_area required non-empty at project+micro-area precision; VIII: bun:sqlite direct, zero new deps).

### ADR-0025 — Article VI "adjacent recommendation" operationalized: block-scoped scanner with a figure-token grammar (T015)
- Date: 2026-07-14
- Zone: YELLOW (interpreting an ambiguous requirement — the test's definitions will judge T016's monthly report too)
- Context: Article VI forbids a zero/negative metric without an "adjacent" recommendation, but no spec doc defines "adjacent" or which digits count as a "figure" (euro amounts like "250.000 €" and dates like "14.07.2026" contain zeros that are not metrics). T015 must turn the rule into an executable test, not spot-checks.
- Decision: tests/naked-numbers.test.ts contains a structural scanner run over the FULL rendered report: (a) adjacency = same Markdown block (heading line through the next heading) — a recommendation in a different section does NOT excuse a naked zero (pinned by a synthetic-markdown test); (b) a "figure" is a standalone integer token, excluding digits glued to `.`/`,` on either side — this grammar drops dot-grouped euro figures and DD.MM.YYYY dates WITHOUT a content allowlist, so any NEW zero metric added to a template is scanned automatically; a bad figure = token ≤ 0; (c) pairing means the block contains a "Σύσταση:" line, and the block-level assertions additionally pin that the text equals recommendation() output verbatim (real data-derived output, not decorative text); (d) anti-vacuous guard: the fixture scan must find at least one PAIRED bad figure — a scan that found nothing to check proves nothing (mirrors Gate 6c's vacuous-pass guard); (e) the scanner's own teeth are pinned by synthetic-markdown tests (naked zero flagged, naked negative flagged, cross-block recommendation rejected, dates/prices/positive counts ignored), so the rule cannot silently die with the detector. RED verified by mutation (repo-standard for rule-pinning tests, per ADR-0012/0015 precedent): removing the per-unit recommendation line from biweekly.ts → 5 failures naming the exact naked "- Επίσκεψη: 0"/"- Προσφορά: 0" lines; removing the zero-period plain statement → exactly the fully-zero test fails; both mutations reverted, git diff vs HEAD empty on src/.
- Alternatives considered: line-adjacency (recommendation within N lines of the figure — rejected: arbitrary N, and a multi-line unit block with the recommendation as its last line is exactly the intended layout); whole-document adjacency (rejected: one recommendation anywhere would excuse every naked zero — obviously not Article VI's intent, and the cross-block synthetic test pins the rejection); allowlisting known-safe strings like dates/prices (rejected: a template change would silently shrink coverage; the token grammar generalizes); putting the scanner in src/ (rejected: it is enforcement infrastructure, not product code — the report must not depend on its own auditor).
- Reversibility: easy — one test file; the adjacency/grammar definitions can be re-pinned if a future template legitimately breaks them.
- Article-safety: confirmed no Article I–IX violation (VI strengthened into an executable, anti-vacuous test; III untouched — no src/ change at all in this task; the scanner itself is deterministic string processing).

### ADR-0026 — Monthly report shape: 30-day tiled periods, median-€/m² comps target, active-inventory per-unit blocks, offers-pace absorption forecast, comp-source labels (T016)
- Date: 2026-07-14
- Zone: YELLOW (interpreting requirements the spec left open: period semantics for "monthly", the comps→€-target formula ADR-0011 assigned to the report layer, the per-unit universe for a project with sold stock, the absorption-forecast formula, and FR-11 wording for the comps.source enum)
- Context: T016 fixes the five report components but not their definitions: what "monthly" and "previous fixed period" mean before T017's anchoring lands; how the comps merge becomes ONE defensible € figure per unit; whether sold units get a sales recommendation; what a deterministic "absorption forecast" from offer/viewing signals only IS; and how `own_transaction`/`manual_known_sale` render in Greek.
- Decision: (a) Period = the 30 CALENDAR days ending on the as-of day, half-open [start, endExclusive) exactly like the biweekly (ADR-0024 semantics); "previous fixed period" = the 30 days immediately before, sharing exactly one boundary string (adjacent, no gap/overlap — FR-13 pinned by a boundary-event test). Calendar months rejected: they cannot tile from T017's arbitrary `--anchor` (default project.listed_at) and their varying length (28–31d) makes trend deltas incomparable. (b) Comps-based € target (report layer per ADR-0011) = MEDIAN €/m² over the T011a merge rows carrying both sqm and sale price (even count → mean of the middle two), × the unit's own sqm, rounded to the €500 grid (counter() convention); null (→ price recommendation without a figure) when the unit or all comps lack sqm — a target not normalized by size would be indefensible in front of a builder. (c) Per-unit recommendation blocks cover ACTIVE inventory only (status ≠ sold/withdrawn): a "staging refresh" line on a sold unit reads absurd; sold units tell their story in the realization section (v_price_realization), withdrawn units are not being marketed. All-sold/withdrawn and no-units-at-all get distinct plain statements. (d) Absorption forecast = ceil(live units / offers-in-period) months at the current offer pace, from offer/viewing signals ONLY (the Phase-B velocity view is never referenced — pinned by a source-scan test); zero offers ⇒ NO numeric forecast is fabricated — stated plainly + recommendation() (Articles III/VI: an invented viewing→offer conversion coefficient would be estimation, not measurement). (e) comps.source renders via a new labels.ts map (FR-11): own_transaction → "δική μας πώληση", manual_known_sale → "καταγεγραμμένη πώληση περιοχής"; unknown key throws like every T006a map. (f) Article VI mechanics follow ADR-0025: trend deltas render with an ASCII sign (scanner-visible, never typographically hidden); any section whose rendered figures include a ≤0 value carries a verbatim recommendation() line in the SAME block (mutation-verified RED: removing the trend-block pairing → 3 named failures, then reverted). Supporting additions: read-only `priceRealization` + `liveUnitCount` in queries.ts (DB layer owns all SQL) and additive `sqm`/`status` fields on UnitWindowActivity (one shared per-unit query serves both reports).
- Alternatives considered: calendar-month periods (rejected above); mean €/m² (rejected: one outlier comp skews the target the builder will be asked to accept — median is robust and equally deterministic); whole-merge median sale price without sqm normalization (rejected: comparing a 60m² unit to a 100m² sale is not a comp); rendering sold units as per-unit blocks with special "sold" copy (rejected: either leaks a stored status key or duplicates the realization section); forecasting from viewings via an assumed conversion rate (rejected: a made-up coefficient is exactly the "estimated, not computed" number Article III forbids); reusing T015's scanner by exporting it from tests/naked-numbers.test.ts (rejected: cross-test-file coupling and edits to T015's landed file — the 30-line grammar is duplicated in tests/monthly.test.ts with an ADR-0025 reference instead).
- Reversibility: easy — window constants, one pure target function, two read-only queries, one label map, and a template; all pinned by re-pinnable tests; no schema change, no stored-data impact; T017 layers period anchoring on top by choosing the as-of.
- Article-safety: confirmed no Article I–IX violation (III: SQL-only numbers, injected as-of, formatEuro + integer percent, no LLM/network/wall-clock/ICU; IV: no buyer fields anywhere in the output; V: micro_area in header AND comparative heading, comps merge keyed on micro_area; VI: every zero/negative figure block-paired with recommendation(), mutation-verified; VII: own sales auto-count in the comparative, never re-entered; VIII: zero new deps).

### ADR-0027 — Report CLI shape: containing-tile fixed windows, data-derived default as-of, quarterly = 90-day monthly template, name-addressed resolution (T017)
- Date: 2026-07-14
- Zone: YELLOW (interpreting requirements FR-13 leaves open: WHICH fixed period a given run reports, what "as-of" defaults to when Gate 4 bans the wall clock from src/report, what renderer "quarterly" uses, and the CLI grammar/exit conventions)
- Context: T017 fixes the flag set and the invariants (fixed non-overlapping default anchored to --anchor/listed_at, --rolling internal, --as-of reproducible, byte-identical stdout, Greek errors exit 1) but not these four semantics.
- Decision: (a) Fixed mode reports the period CONTAINING the as-of day — periods tile [anchor + k·P, anchor + (k+1)·P) for P = 14/30/90 days, and the report is framed at the tile's canonical boundaries (the renderers already compute "P days ending on asOf", so the CLI selects the tile by passing its last day as the effective as-of; any as-of inside one tile therefore renders byte-identical output, pinned by test). Unclamped: the frame is the full tile even mid-period — with a truthful --as-of the DB cannot contain later events, and clamping would make the "fixed" window elastic. As-of before the anchor tiles backwards (negative k) rather than erroring — deterministic and harmless. (b) Default as-of is DATA-derived, never the wall clock (ZONING Step 3 bans new Date()/Date.now() in src/report outright): the project's latest event day (MAX substr(event_date,1,10) via a new read-only latestEventDay query), else listed_at. Corollary: "same DB + same flags ⇒ identical stdout" holds byte-exactly even with --as-of omitted, across days and machines. (c) quarterly = the T016 monthly template over a 90-day tile (US-6 groups monthly/quarterly on the same extended shape): monthly.ts gains an additive cadence option ("monthly" 30d default / "quarterly" 90d + title "Τριμηνιαία αναφορά προόδου πωλήσεων"), monthlyWindow an optional periodDays, and previousWindow derives its length from the window itself — zero behavior change for existing callers (T016 suite untouched and green). (d) Resolution is by exact builder_name + project_name (new read-only findProject, ties → lowest id; builderExists distinguishes the two Greek errors: «Άγνωστος κατασκευαστής …» vs «Άγνωστο έργο … για τον κατασκευαστή …»). CLI grammar follows the ADR-0021 comps precedent: node:util parseArgs strict, --help → usage exit 0, no args/unknown flag/missing required flag/renderer error → Greek stderr + exit 1, report bytes via process.stdout.write (no console.log newline injection). Tiling mutation-verified (Math.floor → Math.round → 6 named failures incl. the FR-13 exactly-one-window pin; reverted).
- Alternatives considered: reporting the last COMPLETED tile (rejected: an as-of early in a project's first period would select the tile BEFORE listing — an empty, nonsensical frame; containing-tile plus the latest-activity default lands on the period where the work actually is); defaulting as-of to today via the wall clock (rejected: RED by ZONING Step 3, and it would silently break byte-reproducibility of flagless runs); a dedicated quarterly.ts renderer (rejected: a near-verbatim copy of the 200-line monthly template, Article VIII); resolving projects by id flags (rejected: task brief and CLAUDE.md name the flags --builder/--project as names — ids are internal); clamping the fixed window at as-of (rejected: two runs inside one period would cover overlapping sub-windows, reintroducing the double-count FR-13 exists to kill).
- Reversibility: easy — window selection is one pure exported function (fixedPeriodEnd) + one effective-as-of expression; the three support queries are read-only; the monthly cadence option is additive; all pinned by 31 re-pinnable tests in tests/cli.test.ts.
- Article-safety: confirmed no Article I–IX violation (III: no LLM/network/wall-clock/ICU — reference day is injected or data-derived, spawn test compares stdout bytes across two runs; IV: no buyer fields touched, buyer_identity never referenced; V: untouched — renderers keep micro_area headers; VI: untouched renderers, naked-numbers suite still green; VIII: zero new deps, one new module).

### ADR-0028 — Insight-brief data shape: JSON with English camelCase keys, cold = zero in-window offers over active inventory, per-unit verbatim recommendation(), one-decimal offer-gap percent (T018)
- Date: 2026-07-14
- Zone: YELLOW (ZONING Step 4: a data shape the spec left OPEN — tasks.md names the signals "cold units, activity deltas, offers-vs-asking gaps … structured text/JSON" but fixes neither the serialization, the key language, the cold-unit definition, the unit universe, nor the gap math)
- Context: T018's brief is the hand-off artifact between the deterministic app and the HUMAN-run /insights step (Article III / ADR-0005). It must be unambiguous for an AI reader, byte-deterministic, and must never let a bad signal travel without its data-derived recommendation.
- Decision: (a) Serialization = JSON (2-space indent, trailing newline, fixed construction order ⇒ JSON.stringify is byte-stable; well-formedness is machine-checkable via JSON.parse in tests, unlike free-form text). Keys are English camelCase like the HTTP API wire format (ADR-0017): the brief is an operator/tooling DATA surface, not the builder-facing Greek report — FR-11 governs rendered stored-enum keys, and the brief renders none (unit status is used only to filter; period cadence echoes the CLI flag vocabulary). Greek prose appears only as recommendation() output. (b) Shape = { project{builderName,projectName,microArea}, period{cadence,start,end,days}, totals, previousPeriod{start,end,…}, deltas, units[], coldUnits[], compsCount } — deltas are signed current−previous over the ADJACENT equally-long previous window (previousWindow(), FR-13 tiling), covering "activity deltas"; compsCount lets the /insights prompt obey its own "never invent comps" rule. (c) Cold unit = ZERO in-window offers, computed over ACTIVE inventory only (status ≠ sold/withdrawn, the T016/ADR-0026 universe); the viewings count on the same row lets the AI step distinguish price-problem cold (≥3 viewings) from presentation-problem cold, and EVERY unit row carries the verbatim recommendation({viewings, offers, compsTarget}) string — Article VI's pairing discipline applied to the brief even though it is not client-facing. (d) Offers-vs-asking gap = { amount: askingCurrent − latestOfferAmount, pctBelowAsking: one-decimal percent via Math.round(x·1000)/10 } only when the LATEST in-window offer (ADR-0013 live position, never MAX) sits below a positive asking; null otherwise. Percent (not the counter() fraction) because the brief's reader is prose-generating — "10" reads as 10%, 0.1 invites a 100× error; one-decimal integer rounding keeps it ICU-free and byte-stable. compsTarget reuses T016's compsBasedTarget (median €/m² × unit m², €500 grid) — one formula across report and brief. (e) CLI: --brief on the existing command reuses the T017 window resolution verbatim (fixed containing tile / --rolling / --as-of / data-derived default), so the brief always describes exactly the window the report covers; errors and exit codes unchanged. (f) .claude/commands/insights.md input-format section updated to document this exact shape (the task-sanctioned touch).
- Alternatives considered: Greek keys or a Greek prose/text brief (rejected: harder to parse deterministically in tests, invites paraphrase drift in the AI step, and diverges from the ADR-0017 camelCase data-surface convention); cold = zero viewings AND zero offers (rejected: a unit with 5 viewings and no offer is precisely the "τιμή ψηλά" signal the mandate conversation needs — offers are the moat metric, and viewings on the row preserve the distinction); including sold/withdrawn units (rejected: a "cold" flag on a sold unit is noise; T016 already ruled non-active inventory out of recommendation blocks); counter()-style fraction for the gap (rejected above); MAX(offer) for the gap (rejected: ADR-0013 — a lower re-offer is the live position); a separate brief CLI command (rejected: Article VIII minimal surface; the report command already owns name resolution + window semantics).
- Reversibility: easy — one new module (src/report/brief.ts), one additive flag branch in cli.ts, and the command-file doc section; all pinned by 23 re-pinnable tests in tests/brief.test.ts; no schema change, no stored-data impact.
- Article-safety: confirmed no Article I–IX violation (III: NO LLM anywhere — the brief is the deterministic half of the ADR-0005 split; SQL-only numbers, injected as-of, no wall-clock/ICU, byte-identical spawn test; IV: no buyer fields — not even pseudonyms — in the brief; V: microArea in the project block; VI: every zero/cold unit row carries recommendation() verbatim, and the client-facing reports are untouched (naked-numbers suite green); VIII: zero new deps).

### ADR-0032 — B0a team-PIN session shape: POST /login, in-memory UUID tokens, HttpOnly session cookie, static-shell exemption, 4-12-digit PIN policy (RULING 2026-07-14b implementation)
- Date: 2026-07-14

### ADR-0033 — B0b HTML deliverable design: escape-first subset renderer, single-document shell with inline system-ui CSS, CLI-level wrap, --html/--brief exclusivity (RULING 2026-07-14b implementation)
- Date: 2026-07-14
- Zone: YELLOW (the ruling sanctions "Markdown → optional HTML" and binds zero deps/schema untouched; the document's structure, CSS approach, and escaping policy were left open)
- Context: builders receive the report as a styled self-contained HTML file sent by email/Viber (RULING 2026-07-14b). Open: how Markdown becomes HTML with zero dependencies, what the document/CSS looks like, how user-entered text (next_action class) is made XSS-safe, and how --html composes with the existing CLI modes.
- Decision: (a) STRUCTURE — src/report/html.ts is a hand-rolled renderer for EXACTLY the Markdown subset the report renderers emit (enumerated from biweekly.ts/monthly.ts/separation.ts: #/##/### headings, **bold**, "- " lists, | pipe | tables with a | --- | separator row, fully-wrapped _italic_ lines, single-line HTML comments, paragraphs); htmlDocument() wraps the body in one fixed shell — <!doctype html>, <html lang="el">, charset + viewport meta, <title> = first H1 text (escaped; fixed Greek fallback), <main> content.
